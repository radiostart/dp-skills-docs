# 도메인 rules 작성

!!! info "한 줄 요약"
    `rules/{domain}.md` — 도메인의 *정책·제약* 을 한 곳에 모아 에이전트가 참조하게 하는 파일입니다. `discover` 가 빈 스켈레톤을 만들어 주고, **내용은 팀이 채웁니다**.

## scope 와 rules — 무엇이 다른가

도메인 컨텍스트의 기본 카테고리는 `scope` 와 `rules` 두 가지입니다. 둘 다 `scope/{domain}.md`·`rules/{domain}.md` 경로면 플러그인이 자동 로드하는 컨트랙트지만, 성격이 정반대입니다.

| 카테고리 | 담는 것 | 성격 | 누가 만드나 |
|---|---|---|---|
| `scope` | 라우트·모델·서비스 — 코드의 *사실* | 코드에서 기계적으로 추출 가능 | [`/dp-skills:discover`](domain-discover.md) 가 내용까지 자동 생성 |
| `rules` | 메모 포맷·상태 전이·layering 규칙 — 도메인의 *정책* | 코드만 봐선 추출 불가 | `discover` 가 빈 스켈레톤만 생성 · **내용은 팀이 작성** |

`discover` 가 `rules` 의 *내용* 까지 만들지 못하는 이유가 여기 있습니다. "주문 service 는 OrderRepository 만 의존한다", "메모 문자열은 `[{액션}({필드})]` 포맷" 같은 규칙은 코드 어디에도 명시돼 있지 않습니다 — 팀의 약속이라 사람이 적어야 합니다. 그래서 `discover` 는 파일 자리(스켈레톤)와 `MANIFEST.md` 의 `rules` 칸만 잡아 주고, 정책 자체는 추출하지 않습니다.

## 언제 쓰나

- `discover` 로 `scope` 를 만든 뒤, 그 도메인의 **비즈니스 규칙·제약** 을 에이전트가 따르게 하고 싶을 때
- 같은 규칙이 `generator.md`·`evaluator.md` 등 여러 곳에 복사돼 **sync drift** 가 우려될 때 — 규칙을 `rules` 한 곳에 두고 나머지는 참조만 하게

## 전제

- 활성 팀 workspace 와 `MANIFEST.md` 가 있을 것
- 대상 도메인이 `MANIFEST.md` 의 `## 도메인 분류` 표에 등록돼 있을 것

## 절차

### 1. 스켈레톤 확인

`discover` 를 거친 도메인이라면 `workspace/{TEAM}/context/rules/{domain}.md` 스켈레톤과 `MANIFEST.md` 의 `rules` 칸이 **이미 만들어져** 있습니다. 스켈레톤은 이런 모양입니다.

```markdown
# checkout 도메인 규칙

> 도메인의 정책·제약을 선언적으로 작성한다 (코드 스니펫이 아니라 정책 표현으로).
> discover 는 이 스켈레톤만 생성한다 — 규칙 내용은 팀이 채운다.

_(추가 예정)_
```

`discover` 를 쓰지 않은 도메인이라면 직접 만듭니다 — `rules/{domain}.md` 파일 생성 + `MANIFEST.md` `## 도메인 분류` 표의 `rules` 칸에 경로 기입. `_(추가 예정)_` 한 줄만 둬도 `/dp-skills:project` 는 동작합니다.

### 2. 내용 구성 — 선언적 정책으로

규칙은 **코드 스니펫이 아니라 정책 표현** 으로 적습니다. 구체 문자열(메모 포맷 등)은 **이 파일 한 곳에** 두고, 에이전트 파일은 "이 규칙 확인" 으로 참조만 하게 합니다.

```markdown
# checkout 도메인 규칙

- 주문 service 는 OrderRepository 만 의존한다 (다른 Repository 직접 호출 금지).
- 결제 메모 문자열은 `[{액션}({필드})]` 포맷을 따른다.
- 환불은 paid·partially_refunded 상태에서만 허용된다.
```

한 줄씩 손으로 적어도 되지만, **AI 프롬프트로 초안을 만들고 다듬는** 방법이 빠릅니다 — 아래 **프롬프트로 rules 초안 만들기** 절을 참고하세요.

### 3. 채워 나가기 — 제안 기반 갱신

처음부터 완성할 필요는 없습니다. 코드 작업 중 에이전트가 "이 규칙이 필요/어긋남" 을 발견하면 직접 고치지 않고 사용자에게 수정안을 보고합니다 — 승인하면 반영됩니다. (`_(추가 예정)_` 플레이스홀더를 처음 채우는 경우만 승인 없이 반영, 출처 주석 필수.)

## 프롬프트로 rules 초안 만들기

도메인 규칙을 한 줄씩 손으로 적는 대신 AI 프롬프트로 초안을 만들면 빠릅니다. 품질은 **프롬프트에 무엇을 담느냐** 로 갈립니다.

### 좋은 프롬프트의 3가지 재료

1. **출력 대상** — 어느 파일에 쓸지 명시 (`rules/{domain}.md` 경로).
2. **원천 지식** — AI 가 규칙을 끌어낼 소스. 아래 중 하나 이상을 줍니다.
    - 팀이 이미 아는 규칙 메모 (비형식이어도 됨)
    - 같은 도메인의 `scope/{domain}.md` (도메인 구조 파악용)
    - 실제 코드 영역 (관찰된 패턴에서 규칙 후보 도출)
3. **형식 제약** — "선언적 정책 표현으로, 코드 스니펫 금지", "한 줄에 한 규칙" 처럼 출력 형식을 명시.

### 상황별 프롬프트 패턴

**① 머릿속 메모를 형식화** — 규칙은 아는데 정리가 안 됐을 때.

```text
workspace/myteam/context/rules/checkout.md 를 다음 팀 규칙으로 채워줘.
선언적 정책 표현으로, 코드 스니펫 말고 한 줄에 한 규칙:

- 주문 서비스는 레포지토리 통해서만 DB 접근
- 환불은 결제완료나 부분환불 상태에서만 가능
- 결제 메모는 대괄호 안에 액션과 필드 적는 포맷
- 쿠폰은 주문당 하나만
```

**② scope·코드에서 초안 추출** — 어떤 규칙이 있는지 잘 모를 때.

```text
scope/checkout.md 와 app/services/checkout/ 를 읽고 rules/checkout.md
초안을 만들어줘. 단 — 코드에 실제로 관찰된 패턴만 적고, 추측한 규칙은
넣지 마. 확신이 없는 항목은 "검토 필요:" 접두어로 표시해줘.
```

**③ 기존 rules 검토·보강** — 이미 채워진 파일을 점검할 때.

```text
rules/checkout.md 를 읽고 — 모호하거나, 서술이 장황하거나, 서로
모순되는 규칙이 있으면 지적해줘. 파일은 수정하지 말고 제안만 해줘.
```

### 결과 검토 체크리스트

AI 초안을 그대로 커밋하지 말고 확인합니다.

- [ ] **정책 표현인가** — 코드 스니펫·함수명 나열이 아닌 규칙 문장인가
- [ ] **추측이 단정으로 섞이지 않았나** — 특히 ② 처럼 코드에서 추출했을 때
- [ ] **한 규칙이 한 줄로 명확한가** — 한 줄에 여러 규칙이 뭉쳐 있지 않은가
- [ ] **scope 와 중복되지 않았나** — 라우트·모델 같은 *사실* 은 `scope` 소관, rules 에 복사 금지

### AI 가 자주 틀리는 것

- **코드를 그대로 옮겨 "규칙"이라 부름** → "정책으로 추상화해줘, 코드 말고" 를 명시.
- **본 적 없는 규칙을 그럴듯하게 지어냄** → "관찰된 것만, 추측은 `검토 필요:` 로 표시" 를 프롬프트에 박아 둠.
- **기존 규칙을 통째로 재작성** → ③ 처럼 "수정 말고 제안만" 으로 범위를 제한.

!!! tip "스켈레톤이 비어 있을 때만 자유 편집"
    `rules/{domain}.md` 가 `_(추가 예정)_` 스켈레톤 상태면 위 ①·② 패턴으로 자유롭게 채워도 됩니다. 이미 팀 규칙이 들어 있는 파일은 ③ 패턴(제안만) 으로 점검하고, 실제 수정은 아래 **흔한 실수** 의 제안 기반 원칙을 따르세요 — 기존 규칙을 임의로 덮어쓰지 않기 위함입니다.

## 흔한 실수

- **코드 리뷰의 `rules.md` 와 혼동하지 않습니다.** `code-review/rules.md` 는 `[severity][grep:패턴]` 4요소 형식의 코드 리뷰 룰입니다. 여기서 다루는 `rules/{domain}.md` 는 도메인 비즈니스 규칙으로, 형식이 자유로운 선언적 정책입니다 — 별개 파일·별개 개념입니다.
- **도메인 규칙을 프로젝트 `agents/*.md` 에 박지 않습니다.** 메모 문구·상태값 리스트를 `generator.md`·`evaluator.md` 양쪽에 복사하면 `rules` 와 drift 가 납니다. SSOT 는 `rules/{domain}.md`, agent 파일은 참조만.
- **에이전트가 context 지식을 직접 수정하지 않습니다.** 발견 시 기록·보고 → 사용자 승인 후 Edit. (3번 절차 참고.)
